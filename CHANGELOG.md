## [0.7.0] - 2026-06-16

### Added

- Recursive loop: the MLX agent provider auto-resolves the model the harness trained. When `AUTOCONTEXT_AGENT_PROVIDER=mlx` and no explicit `AUTOCONTEXT_MLX_MODEL_PATH` is set, `build_client_from_settings` now looks up the active MLX model in the model registry for the scenario (training auto-activates published models), so a run uses the local model a prior run produced and activated. Mirrors the existing `pi` handoff resolution. An explicit `AUTOCONTEXT_MLX_MODEL_PATH` still takes precedence; only `mlx`-backend full checkpoints (servable by `MLXProvider`) are resolved; a clear error is raised when neither a path nor an active model is available.

- Case study: validated training result. `autocontext/docs/case-study-on-policy-distillation.md` documents an end-to-end GPU validation of the cross-platform `trl` backend on GSM8K: a 1.5B student distilled from a 3B teacher closed ~30% of the teacher gap at 1000 steps (0.703 +/- 0.021 over 3 seeds) and ~43% at 2000 steps, while RLVR (GRPO) stayed flat at matched compute, reproducing the on-policy-distillation result on autocontext's own scenario + verifier abstraction. Also adds a `--seed` flag to `autoctx train` (threaded through the trl backend) for seeded repeats.

- Cross-platform TRL backend (`--backend trl`, non-Apple). The MLX `grpo` and `opd` backends are Apple-Silicon only; this is their cross-platform counterpart, wrapping HuggingFace TRL's validated trainers so the same two methods run wherever `trl` + `torch` are installed (Linux / NVIDIA / CPU) -- where a real scale/efficiency-validation run belongs. `--trl-mode gkd` runs on-policy distillation via TRL `GKDTrainer` (the off-the-shelf equivalent of `opd`; `lmbda=1.0` fully on-policy, `beta=1.0` reverse KL); `--trl-mode grpo` runs RLVR via `GRPOTrainer`, reusing the **same** tested `score_completions` reward adapter as the MLX `grpo` backend. The module keeps the numerics in TRL, so its config / dataset / reward seams have no heavy imports and are unit-tested directly (the trainer-instantiating runner imports `trl`/`torch`/`peft` lazily); a `--time-budget` callback stops training at the next step boundary, and a tokenizer-vocab mismatch is rejected up front. `--base-model` is the student and `--teacher-model` (gkd) the teacher (HuggingFace repo ids; must share a tokenizer); LoRA via PEFT. `TRLBackend` is registered on both runtimes. Install with `uv pip install trl peft`. See `docs/mlx-training.md`.
- On-policy distillation backend (`--backend opd`, Apple Silicon). A dense-reward distillation backend: the student samples completions on-policy and is trained to match a frozen teacher's per-token distribution via reverse KL `KL(student || teacher)` over the student-generated positions (Agarwal et al. GKD / Thinking Machines on-policy distillation). Reverse KL is mode-seeking and unhackable, and the per-token signal is far denser than RLVR's one reward per episode. `mlx-lm-lora` has no GKD mode, so this is an MLX-native in-process trainer: a differentiable reverse-KL loss kernel (`reverse_kl_per_token`, teacher stop-gradient), an on-policy rollout + step (`sample_completion`, `on_policy_distill_step`), and a training loop (`distill_over_prompts`) that loads a frozen teacher + LoRA student, distills over scenario-derived prompts, and saves a reloadable adapter. `--base-model` is the student; the teacher defaults to a larger same-family model (they must share a tokenizer). The TypeScript orchestration registry gains a matching `opd` entry. Needs `mlx-lm`. For cross-platform / larger runs the off-the-shelf counterpart is TRL's `GKDTrainer`. See `docs/mlx-training.md`.
- R1 recipe end-to-end (`autoctx train-r1`). Chains the two training backends into the R1-style recipe: a reasoning distillation cold-start (the `mlxlm` backend) followed by GRPO/GSPO RLVR (the `grpo` backend) that **resumes from the distilled adapter** (`--resume-adapter-file`) rather than restarting from the base model. `run_r1_pipeline` runs distillation into `<output-dir>/distill/` then RLVR into `<output-dir>/rlvr/` resuming the cold-start adapter (falling back to the base model if the distill stage produced none), with both stages sharing one base model so the resume is valid. `--variant` selects the RLVR algorithm (`gspo` | `grpo` | `dr_grpo` | `dapo`) and `--register-import` registers a consumer-repo scenario (applied in the parent for the in-process distillation stage and inside the RLVR subprocess). The TypeScript orchestration backend registry gains matching `mlxlm` and `grpo` entries for parity. Needs `mlx-lm` and `mlx-lm-lora`. See `docs/mlx-training.md`.
- GRPO prompt diversity. The GRPO backend now trains over a DIVERSE set of instances instead of one repeated prompt: `build_prompt_rows` derives each row from `scenario.initial_state(seed)` + `get_task_prompt(state)` (serializing the instance state into the `answer` field), and `score_completions` verifies each completion against its own instance state (forwarded by the generated reward function). Parameterized scenarios (whose prompt depends on the state) thus give GRPO real prompt variation to learn a general rule from; fixed scenarios are unchanged (identical prompts). This addresses the documented weakness where single-prompt RLVR has little signal to learn from.
- GRPO/GSPO RLVR training backend (`--backend grpo`). Online RL from the scenario verifier: instead of using the reward offline (filter/condition/weight an SFT set), it has mlx-lm-lora sample a group of completions per prompt, scores each with the scenario's `execute_match`/`evaluate_output` (no labelled answers needed), and takes a GRPO-family policy-gradient step. `score_completions` is the reward adapter (robust to reason-then-construct output via `extract_json_object`); variants map to mlx-lm-lora flags: `gspo` = sequence-level importance sampling (default, the Qwen GSPO stability fix), `dr_grpo` = Dr.GRPO loss, `dapo` = clip-higher, `grpo` = token-level baseline. Gated behind `mlx-lm-lora` (install: `uv pip install mlx-lm-lora`; a packaged `grpo` extra + lock bump is a follow-up). Best paired with a capable base and an in-reach scenario (small/weak bases hit a documented RLVR capability ceiling).
- Teacher-reasoning distillation (provider-agnostic; all opt-in, defaults byte-identical). A gated `<|reasoning|>` segment in the training sequence format lets a record carry the rationale that produced its construction, and completion-only loss covers the reasoning and the strategy so the model learns to reason and then construct. A new trace collector (`training/autoresearch/trace_collector.py`) prompts any `LLMProvider` (hosted, local / OpenAI-compatible, or a callable) to reason and emit a construction, verifies the construction in-scenario, and keeps only score-gated traces as training records. The mlx-lm backend threads the rationale into its completions and record curation preserves it, so a record with a `reasoning` field trains reason-then-construct end-to-end. Absent reasoning, the format, completions, and loss mask are unchanged.
- autoresearch generation sampling controls. `assess_strategy_quality` and the local-training generator (`prepare._generate_strategy_text`) now accept `temperature`, `top_k`, and a per-sample seed, threaded through `run_training` / `_run_mlx_training` and exposed as `--assess-temperature` / `--assess-top-k` on the train CLI. The default is `temperature=0.0` (greedy and deterministic, byte-identical to prior behavior), so diverse assessment is opt-in: pass a positive temperature to draw varied samples and let the oracle measure generation diversity. The CUDA backend (`run_cuda_training`) threads the same controls through its torch generator for parity.
- autoresearch training-data pipeline improvements (all opt-in; defaults reproduce prior behavior):
  - Completion-only loss + document-boundary packing: loss is computed only on the strategy completion (tokens after `<|strategy|>`), examples are batched per-document with no cross-example contamination, and the final partial batch is no longer dropped.
  - Held-out validation: `autoctx train` reports a `val_loss`, and `--val-select` keeps the best-by-validation-loss checkpoint and early-stops (MLX backend only).
  - Record curation: `--elite-fraction <0-1]` trains on only the top fraction of records by score, `--dedupe` drops duplicate constructions (keeping the highest-scoring representative), and `--dedupe-near-threshold <0-1]` additionally removes near-duplicates by character-shingle similarity. Out-of-range values are rejected before training; the published model artifact records the curation settings plus raw and curated record counts. See `docs/mlx-training.md`.
  - Tokenizer vocabulary size: `--vocab-size` (default 8192) sets the BPE tokenizer's target vocab for the from-scratch `mlx` / `cuda` backends, threaded through `train_tokenizer`; the model head/embedding auto-sizes to the resulting tokenizer vocab (base + structural special-token slots), so a smaller vocab trades longer sequences for more subword sharing and a larger one shortens sequences. Validated to be `>= 256` (the byte-level BPE base). The `mlxlm` backend rejects a non-default value with a clear `NotImplementedError` (it uses the pretrained model's tokenizer); the chosen size is recorded in `data_stats`. See `docs/mlx-training.md`.
  - Symmetry / transform augmentation hook: `--augmenter "package.module:function"` resolves a domain record-augmenter by dynamic import and applies it to the training records before curation, so a domain symmetry/transform (e.g. affine orbits for cap-sets) can multiply effective training data while autocontext core stays domain-agnostic (the transforms live in the consumer repo). The augmenter is a `list[record] -> list[record]` callable; a malformed spec or non-list/empty result fails fast. Applies to all backends (mlx/cuda/mlxlm); the spec is recorded in `data_stats`. New `prepare_training_records` (augment + curate) seam + `augment.py` (`resolve_augmenter`, `apply_augmentation`). See `docs/mlx-training.md`.
  - Pretrained fine-tuning backend (`--backend mlxlm`, `mlxlm` extra): LoRA/DoRA fine-tunes a pretrained `mlx-lm` model (default `mlx-community/Qwen2.5-0.5B-Instruct-4bit`) on the curated, optionally score-conditioned records instead of training a from-scratch GPT, using the base model's tokenizer + completion-only loss (`--mask-prompt`). New `--base-model` / `--fine-tune-type` (`lora`/`dora`/`full`) / `--num-layers` options. Reuses record curation + the scenario interface; score-conditioning is expressed as a natural-language quality directive. See `docs/mlx-training.md`. (The `train` command moved to `cli_train.py` to keep `cli.py` within its size budget.)
  - Score-conditioned generation: `--score-conditioned` adds a `<|quality|>` control token before the strategy (derived from the record's score, quantized into buckets over `[0, 1]`) and conditions assessment/inference generation on the top quality bucket (Decision-Transformer / Quark style). The token is **gated**: it is reserved/registered only for score-conditioned runs (id appended last), so the default model vocab and architecture are unchanged. The conditioning contract (`score_conditioned`, `num_quality_buckets`) is persisted in the checkpoint `config.json` and registry `data_stats`, and `MLXProvider` reapplies the top-bucket prompt when serving a score-conditioned checkpoint. See `docs/mlx-training.md`.
  - Reward-weighted loss (reward-weighted regression): `--loss-weight-by-score` (`uniform` | `linear` | `softmax`) scales each training example's loss by a weight derived from its score, the soft counterpart to elite filtering (keep every record, but lean the gradient toward high-reward constructions). The weight is applied **per example** (each example's mean completion loss is weighted, then averaged), returned as a separate per-example weight vector rather than folded into the per-token mask, so a long completion cannot drown out a short high-reward one. Non-uniform modes are mean-normalized to 1.0 (they change relative emphasis, not the overall step size); `--loss-weight-temperature` (must be `> 0` for `softmax`) is a continuous knob between uniform and soft-elite; no score spread falls back to uniform. `uniform` (default) is byte-identical to prior training. Applies to the `mlx` and `cuda` backends (the `mlxlm` backend rejects non-uniform modes); the setting is recorded in the registry `data_stats`. Pure helper `score_loss_weights` lives in `sequence_format.py`. See `docs/mlx-training.md`.
  - Self-improving loop (ReST-EM / expert iteration): new `autoctx self-improve` command runs an outer training loop. Each round trains on the current dataset, samples constructions from the trained model and scores them in-scenario, keeps the top `--elite-fraction` by score, and appends them as new training records, so the model generates its own training data biased toward the best of what it can produce (the iterated form of rejection-sampling fine-tuning). Because a round trains before appending its own elite, the loop runs one final training pass over the full accumulated dataset (into `<output-dir>/final`, reported as `final_avg_score`) so a model that has seen every collected sample exists. The final all-data pass can overfit and regress below an earlier round, so the loop selects the **highest-scoring pass** across all rounds and the final pass as the deployable artifact: `best_model_dir` (with `best_round` = `"round_N"` or `"final"` and `best_avg_score`) is the model to ship, and `final_model_dir` is retained only for compatibility/inspection (the final pass is chosen as best only when it actually scores highest). Generated elite records inherit the seed dataset's representative `context` so every training example shares the same context prefix instead of mixing rich seed context with empty prefixes. Sampling uses a positive temperature so collected samples are diverse. Carries `--score-conditioned` through every round and prints a per-round table (avg_score, samples, elite, dataset size) that highlights the best model to ship; `--json` returns the structured `history` plus `best_model_dir` / `best_round` / `best_avg_score` (deployable) and `final_model_dir` / `final_avg_score` (compatibility/inspection). Loop counts (`--rounds` / `--samples-per-round` / `--train-steps`) are validated as positive integers. Sample collection is MLX-only (`run_training(collect_samples_path=...)`); the cuda/mlxlm backends reject it with a clear `NotImplementedError`. Pure helpers (`select_elite_samples`, `samples_to_records`, `representative_context`) live in `training/autoresearch/self_improve.py`. See `docs/mlx-training.md`.

### Changed

- The baseline MLX GPT model (`ModelConfig`, `GPTModel` + components, `compute_loss`, and checkpoint I/O `save_checkpoint` / `load_checkpoint` / `save_inference_bundle`) was extracted from `training/autoresearch/train.py` into a new `training/autoresearch/model.py`, separating the (serving-shared, stable) model architecture from the training-loop orchestration. Behavior is unchanged: every name is re-exported from `train.py` so existing `from ...train import` paths keep working, and `MLXProvider` now imports the model directly from `model.py` (the serving path no longer reaches into the training-loop module). The model-shape knobs stay tunable from `train.py` via `MODEL_DEPTH` / `MODEL_ASPECT_RATIO` / `MODEL_HEAD_DIM` / `MODEL_N_KV_HEADS` constants (which override the `ModelConfig` defaults at construction), so the training-agent revision loop keeps editing model shape; the deterministic variant regex-tunes those constants and the LLM agent prompt includes `model.py` as read-only architecture context. This keeps `train.py` within the 800-line module-size budget.

### Fixed

- autoresearch assessment generation collapsed on structured-output scenarios. Greedy argmax with no per-sample seed produced N identical completions, and for JSON-style strategies it decoded into the most frequent special token, yielding `valid_rate=0`. A shared, pure `generation_logit_mask_values` now blocks both the undecodable phantom-id gap (when the BPE learns fewer merges than `base_vocab_size`, sampling those ids previously raised `KeyError: Invalid token for decoding`) and the structural `<|scenario|>` / `<|context|>` / `<|strategy|>` tokens. The mask is applied in both the greedy and sampling paths and is reused by `MLXProvider` inference so saved bundles cannot sample an undecodable id at serving time.
- the `mlx` optional-dependency extra was missing `numpy`, so `save_checkpoint` crashed at the end of every local training run. The extra now installs `mlx`, `rustbpe`, `tiktoken`, `safetensors`, and `numpy`, and the lockfile is regenerated to match.
- `run_training` now runs a backend dependency preflight inside each backend entry, failing fast with an actionable install hint (`uv sync --group dev --extra mlx`) instead of crashing partway through a run when a backend dependency is absent.

- Expanded the research surface with validated domain contracts, runtime gating, persistence hardening, and better evaluation wiring for briefs, prompts, and adapters.

## [0.3.2] - 2026-04-02

- Hardened the TypeScript operator control plane so terminal session and worker states stay terminal, remote approvals require connected controllers, and redirected work remains visible in progress summaries.
- Python and TypeScript package metadata are bumped to `0.3.2`.

## [0.3.0] - 2026-03-29

### Added

#### Commands

- **`autoctx simulate`** — plain-language multi-variable simulation with sweeps, replay, compare, and export.
- **`autoctx investigate`** — evidence-driven diagnosis with hypotheses, confidence scoring, and unknowns.
- **`autoctx analyze`** — interpret and compare runs, simulations, investigations, and missions.
- **`autoctx train`** — train distilled models from curated datasets with backend selection.
- **Python `autoctx simulate`** — full parity with the TypeScript surface: run, replay, compare, and export.

#### Scenarios

- All 11 scenario families now fully executable in TypeScript (was 2/11) via secure-exec V8 isolate codegen.
- `operator_loop` is now a fully runnable family in both packages.
- Unified family classifier: all families reachable through the CLI.
- Spec auto-heal: codegen failures trigger automatic recovery.
- Scenario revision flow: refine created scenarios with feedback.
- Deep execution validation: generated code is executed and verified before registration.
- Three scenario templates: content-generation, prompt-optimization, and rag-accuracy.
- `new-scenario` CLI materializes runnable artifacts to disk.
- Scenario parity matrix documents Python/TypeScript surface coverage.

#### Missions & Campaigns

- Adaptive mission execution: LLM-driven goal decomposition and step planning replaces generic bookkeeping.
- Campaign abstraction: coordinate multiple missions under long-term goals with budget tracking and dependencies.
- Mission-simulation integration: missions invoke simulations as planning tools.

#### Trace Pipeline

- Open public trace schema v1.0.0: versioned interchange format for coding agent traces.
- Sensitive-data detection and redaction with policy-backed actions.
- Privacy-aware trace export workflow: redact, validate, manifest, and attestation.
- Publishing connectors for local JSONL, GitHub Gist, and Hugging Face.
- Trace-to-model data plane with `DatasetCurator` and `DataPlane`.
- Repo-local dataset discovery: scan repo trees and convert JSONL, JSON, CSV, and markdown into ShareGPT-style records.
- Curated distillation dataset pipeline with gate filtering, top-quartile selection, family filtering, and failure-example policy.

#### Training & Distillation

- Base model selection maps scenario families to training modes (from-scratch, LoRA, and full fine-tune).
- Training backend abstraction with MLX and CUDA plus an injectable `TrainingExecutor` hook.
- Prompt alignment ensures distilled models match runtime invocation.
- Candidate-shadow-active promotion lifecycle with configurable quantitative gates and rollback.

### Changed

- Consolidated operator UI: the Python `serve` and `tui` surfaces are API/WebSocket-first, while interactive terminal UI remains available through the TypeScript client surfaces.
- Richer sweep DSL: categorical sweeps, logarithmic scales, sweep file loading, and named presets.

### Fixed

- Trace pipeline audit: expanded redaction patterns, ISO 8601 timestamp validation, explicit role mapping, export warnings, and Hugging Face format fixes.
- Distillation audit: training executor hook, base model validation, CSV parser edge cases, silent catches now surfaced as warnings, and end-to-end integration coverage.

## [0.2.4] - 2026-03-26

### Added

- Session notebook context now flows into runtime prompts and cockpit views for active runs.
- World-state abstractions now support stateful scenario families and workflow-style scenarios.

### Changed

- Agent-task scaffolding and execution now use separate phased budgets.
- Operator-loop scenarios remain available as typed family metadata, but executable operator-loop scaffolding has been removed so the harness no longer bakes in escalation-specific runtime behavior.
- Public repo docs now include a docs landing page, package-selection guidance, an analytics/adoption guide, a release checklist, and copy-paste integration examples for CLI, MCP, Python SDK, and TypeScript usage.

### Fixed

- Python package fallback version metadata now matches the published `0.2.0` package version.

## [0.2.0] - 2026-03-15

### Added

- Initial public release with Python and TypeScript packages.
- Generation loop with Elo-based progression gating.
- Agent roles: competitor, analyst, coach, architect, and curator.
- Pluggable scenarios including `grid_ctf`, `othello`, and the custom creation pipeline.
- LLM judge with multi-sample evaluation.
- Task runner daemon with improvement loops.
- MCP server with tool implementations.
- FastAPI dashboard with WebSocket events.
- CLI via Typer (Python) and `parseArgs` (TypeScript).
